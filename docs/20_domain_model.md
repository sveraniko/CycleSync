# CycleSync Domain Model

> Domain model document for **CycleSync**.

---

## 1. Purpose of this document

This document defines the high-level domain model of CycleSync.

Its goals are to:

- identify the core domain entities and aggregates;
- define ownership boundaries between bounded contexts;
- describe how contexts reference each other;
- establish naming and modeling rules;
- fix the distinction between compound catalog truth and user/protocol truth;
- prepare the foundation for later data model, event catalog, search model, API contracts and implementation.

This document is **not**:

- a full SQL schema;
- a migration plan;
- a repository design;
- a DTO catalog;
- an API payload catalog;
- a formula-level algorithm document.

Those belong in separate docs later.

---

## 2. Modeling principles

### 2.1 Domain-first, transport-second

The system must be modeled around product/business concepts first, not around Telegram callbacks, ORM convenience or UI shortcuts.

### 2.2 Algorithm-centered modeling

The domain model must preserve the fact that **Pulse Engine** is the central product differentiator.

This means:

- protocol truth and pulse-plan truth must be explicit;
- reminder truth must not replace protocol truth;
- AI and specialist layers must not silently become the “real core” of the system.

### 2.3 Bounded ownership

Every important entity must have a clear owning bounded context.
Other contexts may reference it, but must not silently take over ownership.

### 2.4 Stable identifiers

All important entities must use stable internal IDs.
Display names, search strings, trade labels, Telegram usernames and message text must not be treated as identifiers.

### 2.5 Aggregate integrity

Core invariants must be protected inside aggregates / domain services, not scattered across handlers.

### 2.6 Reference by ID, not by deep object sharing

Cross-context references should generally happen through IDs and contracts, not by embedding uncontrolled domain objects inside other contexts.

### 2.7 Derived views are not domain truth

Search documents, adherence summaries, AI summaries, analytics tables and specialist read-models are derivative views.
They do not define domain truth.

### 2.8 Constrained user freedom

The domain must explicitly reflect the rule that a normal user is **not** allowed to manually control all low-level pharmacokinetic math.

User freedom is constrained through:

- selected compounds;
- weekly target;
- duration;
- preset choice;
- volume/frequency limits;
- specialist-approved plans.

Full manual micromanagement belongs to a later specialist mode, not to the normal user flow.

## 3. Core bounded contexts in the domain model

This domain model is organized around these contexts:

1. Compound Catalog
2. User Registry
3. Protocols
4. Pulse Engine
5. Reminders
6. Adherence
7. Labs
8. AI Triage
9. Expert Cases
10. Search
11. Analytics
12. Integrations (later)

Only some of these own primary transactional truth.
Search and Analytics are largely projection-driven contexts.
AI Triage owns assessments, but not the raw lab truth.

All core domain objects should use stable internal IDs.

Recommended identifier categories:

- `compound_id`
- `brand_id`
- `user_id`
- `telegram_user_id` (external identity reference)
- `protocol_id`
- `protocol_line_id`
- `pulse_plan_id`
- `pulse_plan_line_id`
- `reminder_event_id`
- `reminder_action_id`
- `lab_report_id`
- `lab_marker_id`
- `ai_assessment_id`
- `expert_case_id`
- `expert_message_id`
- `search_document_id` (projection id if useful)
- `analytics_event_id`

### 4.1 Important rule

User-entered values such as:

- brand spelling;
- compound query text;
- Telegram username;
- search query;
- reminder notes;
- uploaded lab filenames

must not be treated as stable identifiers.

## 5. Ubiquitous language

The following concepts should remain stable across the project.

### Compound
A catalog entry representing a concrete product / preparation / formulation available for search and use in protocols.

### Substance
An active pharmacological substance referenced inside compound composition.

### Ester
A modifier / form relevant to pharmacokinetic behavior and scheduling.

### Protocol
A user-associated plan that defines what set of compounds is being used, with what weekly target, duration, constraints and preset.

### Pulse Plan
A computed dosing/injection schedule generated by Pulse Engine to keep the total background as even as possible.

### Preset
A named scheduling strategy such as Unified Rhythm, Layered Pulse, or Golden Pulse / Conveyor.

### Reminder
A scheduled user-facing event derived from a pulse plan or follow-up flow.

### Adherence
The domain truth of whether the user actually followed the protocol event: done, snoozed, skipped, delayed, broken.

### Lab Report
A user-provided analysis package containing one or more markers and optional source attachments.

### AI Assessment
A structured preliminary evaluation of lab data and its changes over time.

### Expert Case
A human-specialist case opened for a user, including protocol, adherence, labs and AI context.

### Search Document
A rebuildable projection optimized for retrieval, not a transactional source of truth.

## 6. Compound Catalog domain model

## 6.1 Purpose

Compound Catalog owns the canonical searchable catalog of preparations and their pharmacokinetic-relevant metadata.

## 6.2 Core aggregate

### `Compound`

Candidate attributes:

- `compound_id`
- `status`
- `display_name`
- `normalized_name`
- `trade_name`
- `brand_id`
- `release_form`
- `packaging_label`
- `primary_media_asset_id` (optional)
- `default_volume_limit_ml` (optional)
- `official_url` (optional)
- `authenticity_note` (optional)
- `created_at`
- `updated_at`
- `source`

A `Compound` represents a concrete searchable product-level entity, not an abstract pharmacology concept.

## 6.3 Supporting entities / value objects

### `Brand`

Candidate attributes:

- `brand_id`
- `display_name`
- `normalized_name`
- `alias_tokens`
- `country` (optional)
- `status`

### `CompoundCompositionEntry`

Represents one active component inside a compound.

Candidate attributes:

- `compound_id`
- `substance_id`
- `ester_id` (optional)
- `amount_mg`
- `amount_unit`
- `concentration_mg_per_ml` (optional depending on form)
- `notes`

This entity is critical because CycleSync search must work not only by trade name, but also by active substance, ester and composition.

### `Substance`

Candidate attributes:

- `substance_id`
- `display_name`
- `normalized_name`
- `alias_tokens`
- `category` (optional)

### `Ester`

Candidate attributes:

- `ester_id`
- `display_name`
- `normalized_name`
- `half_life_days`
- `notes`

### `CompoundAlias`

Purpose:

- alternate spelling support;
- transliteration support;
- noisy copy-paste support;
- search enrichment.

### `CompoundMediaLink`

Links product cards to images/video/reference documents.

### `AuthenticityReference` (later)

Represents instructions or reference material for checking originality.

## 6.4 Invariants

- a compound must have a stable internal ID;
- a compound must belong to a brand or clearly indicate missing brand;
- a searchable compound must be backed by normalized tokens/aliases;
- composition entries must be structurally valid;
- an ester half-life used by scheduling must be explicit and not guessed silently;
- catalog sync must not corrupt existing references.

## 6.5 Compound Catalog does not own

- user profiles;
- user protocols;
- pulse plans;
- reminders;
- adherence;
- lab history;
- specialist cases.

## 7. User Registry domain model

## 7.1 Purpose

User Registry owns the canonical user identity and lightweight operational profile.

## 7.2 Core aggregate

### `User`

Candidate attributes:

- `user_id`
- `telegram_user_id`
- `display_name`
- `username` (optional)
- `status`
- `timezone`
- `language_code` (optional)
- `created_at`
- `updated_at`
- `last_activity_at`

### `UserProfile`

Candidate attributes:

- `user_id`
- `sex` (optional)
- `birth_date` or `age`
- `height_cm` (optional)
- `weight_kg` (optional)
- `goal`
- `notes`

### `UserPreference`

Purpose:

- reminders on/off;
- preferred reminder hour;
- voice mode preference later;
- search/result presentation preferences;
- specialist escalation preferences later.

### `UserLimitProfile`

Purpose:

- max injection volume;
- convenience constraints;
- max preferred frequency;
- later site preferences or route preferences if ever supported.

## 7.3 Invariants

- a user must have a stable internal identity;
- timezone must be present or defaulted before reminder scheduling becomes active;
- preferences and limits must not mutate protocol truth directly;
- user updates must emit projection-refresh relevant events where needed.

## 7.4 User Registry does not own

- catalog data;
- protocol logic;
- pulse-plan math;
- reminder dispatch state;
- lab markers;
- AI assessments;
- expert recommendations.

## 8. Protocols domain model

## 8.1 Purpose

Protocols owns user-associated protocol plans and their business parameters before/after calculation.

## 8.2 Core aggregate

### `Protocol`

Candidate attributes:

- `protocol_id`
- `user_id`
- `status`
- `preset_code`
- `weekly_target_mg`
- `duration_weeks`
- `start_date`
- `max_injection_volume_ml`
- `max_injections_per_week` (optional)
- `created_at`
- `updated_at`
- `confirmed_at` (optional)
- `specialist_locked` (optional later)

### Possible protocol status values

Illustrative only:

- draft
- calculated
- confirmed
- active
- paused
- completed
- cancelled
- archived

## 8.3 Supporting entities

### `ProtocolCompound`

Represents a selected catalog compound inside the protocol.

Candidate attributes:

- `protocol_line_id`
- `protocol_id`
- `compound_id`
- `position`
- `is_required`
- `notes`

Important: for the normal user flow, `ProtocolCompound` is selection truth, not full manual scheduling truth.

### `ProtocolConstraint`

Purpose:

- carry explicit constraint overrides;
- capture user comfort/volume/frequency boundaries;
- future specialist-imposed limits.

### `ProtocolPreset`

Represents the selected scheduling strategy.

Candidate attributes:

- `preset_code`
- `title`
- `description`
- `is_active`

## 8.4 Invariants

- a protocol must reference a valid user;
- a protocol must contain at least one compound before calculation;
- a normal user protocol must not expose uncontrolled low-level scheduling freedom;
- status transitions must remain valid;
- changing core protocol inputs invalidates or supersedes an old pulse plan.

## 8.5 Protocols does not own

- pharmacokinetic calculations;
- reminder scheduling truth;
- adherence truth;
- lab truth.

## 9. Pulse Engine domain model

## 9.1 Purpose

Pulse Engine owns the computed schedule logic required to maintain the flattest feasible background under chosen constraints.

## 9.2 Core aggregate

### `PulsePlan`

Candidate attributes:

- `pulse_plan_id`
- `protocol_id`
- `algorithm_version`
- `preset_code`
- `window_days`
- `flatness_score`
- `estimated_injection_frequency`
- `status`
- `created_at`
- `superseded_by_pulse_plan_id` (optional)

A `PulsePlan` is not the same as a `Protocol`.
It is a computed artifact owned by the engine.

## 9.3 Supporting entities / concepts

### `PulsePlanLine`

Represents one scheduled action or one compound-specific plan entry.

Candidate attributes:

- `pulse_plan_line_id`
- `pulse_plan_id`
- `day_offset`
- `compound_id`
- `dose_mg`
- `volume_ml`
- `sequence_order`

### `PulseWindow`

Represents the calculation window used by the engine.

### `PresetStrategy`

Represents strategy identity such as:

- Unified Rhythm
- Layered Pulse
- Golden Pulse / Conveyor

### `FlatnessScore`

Represents an engine-produced stability metric.
It is a domain-derived score, not a marketing string.

## 9.4 Invariants

- a pulse plan must reference exactly one protocol;
- a pulse plan must be reproducible from protocol inputs and catalog truth;
- pulse plan lines must respect volume and preset constraints;
- recomputation must not mutate past plan truth silently;
- if a plan is replaced, the replacement relation must be explicit.

## 9.5 Pulse Engine does not own

- user identity;
- reminder dispatch history;
- adherence actions;
- raw lab reports.

## 17. Cross-context reference rules

Cross-context relationships must be explicit and disciplined.

### 17.1 Allowed pattern

- Compound Catalog owns `compound_id`
- User Registry owns `user_id`
- Protocols stores `user_id` and selected `compound_id`s
- Pulse Engine stores `protocol_id` and references compounds through protocol context
- Reminders stores `user_id`, `protocol_id`, `pulse_plan_line_id`
- Adherence stores `reminder_event_id` / `user_id`
- Labs stores `user_id` and optional `protocol_id`
- AI Triage stores `lab_report_id`, `user_id`, optional `protocol_id`
- Expert Cases stores `user_id` and contextual links only
- Analytics stores references only
- Search stores projection copies only

### 17.2 Discouraged pattern

- duplicating compound truth inside protocol tables as canonical source;
- duplicating user identity truth inside expert case tables;
- treating search docs as primary data;
- treating AI summary as lab truth;
- letting reminders become adherence truth.

### 17.3 Reference rule of thumb

A context may cache/denormalize selected fields for read optimization, but ownership must remain explicit.

## 20. Domain relationships overview

Simplified conceptual relationships:

- one `User` -> many `Protocols`
- one `Protocol` -> many `ProtocolCompounds`
- one `Protocol` -> one or many `PulsePlans` over time
- one `PulsePlan` -> many `PulsePlanLines`
- one `User` -> many `ReminderEvents`
- one `ReminderEvent` -> zero or many `ReminderDispatch` attempts
- one `ReminderEvent` -> zero or one terminal `AdherenceAction` in v1 (extendable later)
- one `User` -> many `LabReports`
- one `LabReport` -> many `LabMarkerValues`
- one `LabReport` -> zero or many `AIAssessments` over time
- one `User` -> many `ExpertCases`
- one `ExpertCase` -> many `ExpertMessages`
- one `Compound` -> many `CompoundCompositionEntries`
- one `Brand` -> many `Compounds`
- one `Substance` -> many composition entries
- one `Ester` -> many composition entries

The exact relational model belongs in a later data model doc.

## 21. Modeling guardrails

The domain model must avoid the following mistakes.

### 21.1 UI-driven domain leakage

Do not model Telegram screens as core domain objects.
UI state and domain state are different concerns.

### 21.2 Search-as-truth

Search document fields must not be treated as the canonical business record.

### 21.3 AI-as-truth

AI assessment output must not overwrite lab truth or protocol truth.

### 21.4 Reminder/adherence collapse

Reminder state must not be used as a shortcut for actual user compliance truth.

### 21.5 Catalog/user entanglement

Do not merge compound catalog truth with user-specific truth “for convenience”.

### 21.6 Specialist-core inversion

Do not let specialist-case structures quietly become the central domain, displacing Pulse Engine.

## 23. Final domain model statement

CycleSync’s domain model must preserve a clear distinction between:

- catalog truth;
- user truth;
- protocol truth;
- computed pulse-plan truth;
- reminder truth;
- adherence truth;
- lab truth;
- AI-derived assessment;
- specialist-case context;
- search and analytics projections.

The model must be strong enough to support product growth, but simple enough to remain understandable and maintainable.

This document defines the business-object spine of the system.
