# PR W14 / PR4 — Regression, Comparison, and Rollout Confidence Pack for PK V2

## 1) Added validation / runner tooling

Implemented dedicated V2 validation tooling in:

- `scripts/validate_medical_v2_catalog.py`

This script is intentionally separate from legacy `scripts/validate_medical_catalog.py` so V2 workbook validation logic stays deterministic/readable and does not inherit old `medical.xlsx` assumptions.

What the new script does:

1. Validates `medical_v2.xlsx` ingest foundation using V2 ingest parser (`Products` + `Ingredients` + related layers).
2. Reads V2 regression sheets (`PulseScenarios`, `SearchCases`).
3. Runs each pulse scenario in both engines (`v1` and `v2`) through real `PulseCalculationEngine` path.
4. Produces per-scenario outputs with explicit traceability fields:
   - `pulse_engine_version_used`
   - `evaluation_source`
5. Builds consolidated V1-vs-V2 comparison objects per scenario.
6. Runs search regression cases from workbook aliases/ingredients/esters.
7. Generates machine-readable and human-readable artifacts.

## 2) How `medical_v2.xlsx` now drives regression

`docs/medical_v2.xlsx` is now used directly as the scenario/search regression source:

- `Products`, `Ingredients` → pulse/search data model inputs;
- `PulseScenarios` → deterministic scenario execution matrix;
- `SearchCases` → search regression matrix.

This makes the workbook a first-class regression artifact (not just ingest seed input).

## 3) How V1 vs V2 comparison is produced

For each `scenario_id` from `PulseScenarios`, runner executes two passes:

- pass A: `PulseCalculationEngine(pulse_engine_version="v1")`
- pass B: `PulseCalculationEngine(pulse_engine_version="v2")`

Comparison layer then computes:

- `flatness_score_delta`
- `peak_trough_spread_delta` (if available)
- `warning_flags_added_in_v2`
- `warning_flags_removed_in_v2`
- `schedule_same_but_evaluation_changed`
- `strictness_change`
- `status_v1`, `status_v2`

This provides explicit evidence on where V2 behavior differs and whether differences are mostly evaluative vs scheduling-layout changes.

## 4) Generated artifacts / reports

Generated by script defaults:

- `docs/medical_v2_validation_report.json`
- `docs/medical_v2_validation_summary.md`

`medical_v2_validation_report.json` includes:

- ingest validation block;
- scenario outputs for both engines;
- per-scenario V1-vs-V2 comparisons;
- search regression outputs.

`medical_v2_validation_summary.md` includes compact explanation sections:

- scenarios with no material difference;
- scenarios with significant flatness change;
- scenarios where V2 introduced warnings;
- mixed-product scenarios most affected;
- search regression match summary.

## 5) Mixed-product / mode coverage

Scenario runner includes explicit scenario-product matrix covering requested risk zones:

- mixed ester/mixed ingredient paths (`PHARMA MIX 1`, `PHARMA SUST 250 Ampules`),
- tablet product path (`OXANDROLONOS` in constrained scenario),
- single-product baseline,
- multi-product stack paths.

Covered input modes:

- `auto_pulse`
- `total_target`
- `stack_smoothing`
- `inventory_constrained`

## 6) Exact local verification commands

Use from repo root:

```bash
PYTHONPATH=. python scripts/validate_medical_v2_catalog.py \
  --out docs/medical_v2_validation_report.json \
  --summary docs/medical_v2_validation_summary.md

PYTHONPATH=. pytest tests/test_medical_v2_validation_pack.py
```
