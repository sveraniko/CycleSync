## 1. Подтверждённые баги

### BUG-1: Несовпадение формата user_id между settings и draft [CRITICAL]

**Корневая причина.**
Два разных хендлера используют разные реализации `_resolve_user_id()`:

- `draft.py` (а также `labs.py`): `return str(telegram_user_id)` → `"123456789"`
- `settings.py`: `return f"tg:{telegram_user_id}"` → `"tg:123456789"`

**Импакт.**
Когда пользователь нажимает «Protocol status» из панели Settings, система ищет `Protocol.user_id == "tg:123456789"`, но протокол был создан через Draft с `user_id = "123456789"`. Результат: **всегда «no active protocol»**, даже если протокол существует и активен.

**Решение.**
Извлечь единую функцию `resolve_user_id()` в `app/bots/core/user_id.py`. Авторитетный формат — `str(telegram_user_id)` (без префикса `tg:`), так как все протоколы, драфты, лабораторные отчёты и напоминания создаются через Draft/Labs-хендлеры, которые используют именно этот формат. Обновить `settings.py` на использование общей функции.

**Затронутые файлы:**
- `app/bots/core/user_id.py` (новый)
- `app/bots/handlers/settings.py` — заменить `_resolve_user_id`
- `app/bots/handlers/draft.py` — заменить `_resolve_user_id` на общую
- `app/bots/handlers/labs.py` — заменить `_resolve_user_id` на общую

### BUG-2: «Set reminder time» запускает поиск вместо ввода времени [CRITICAL]

**Корневая причина.**
Хендлер `on_set_time_prompt()` (`settings.py`, строка 106) устанавливает FSM-состояние `ReminderSettingsState.reminder_time` и показывает промпт. Однако поисковый хендлер `search_entrypoint()` (`search.py`, строка 26) зарегистрирован с фильтром `F.text` без ограничения по FSM-состоянию. Поскольку `search_router` включается в `router.py` **раньше** `settings_router`, все текстовые сообщения перехватываются поиском.

Текст `09:30`, введённый пользователем как время напоминания, интерпретируется как поисковый запрос.

**Решение.**
Добавить на `search_entrypoint` гард по FSM-состоянию: пропускать сообщение, если у пользователя установлено любое FSM-состояние. Техническая реализация — добавить `StateFilter(None)` или проверку `await state.get_state() is None` в начале хендлера.

**Затронутые файлы:**
- `app/bots/handlers/search.py` — добавить FSM state guard

### BUG-3: Напоминания не отправляются — нет фонового планировщика [CRITICAL]

**Корневая причина.**
`materialize_reminders` и `dispatch_reminders` — это **одноразовые CLI-скрипты** (цели в Makefile). В системе отсутствует фоновая задача, cron, APScheduler или отдельный сервис в `docker-compose.yml`, который запускал бы их периодически.

Цепочка при активации протокола:
1. Создаётся `ReminderScheduleRequest` со статусом `"requested"` в БД
2. **Никто** не вызывает `materialize_requested_schedules()` для создания `ProtocolReminder`-строк
3. **Никто** не вызывает `dispatch_due_reminders()` для отправки Telegram-сообщений

**Решение.**
Добавить периодическую фоновую задачу. Варианты:
- **(A) Рекомендуемый:** APScheduler-петля внутри `run_bot.py`, которая вызывает materialize + dispatch каждые N минут
- (B) Отдельный `scheduler`-сервис в `docker-compose.yml`
- (C) Минимум: при активации протокола — немедленно вызывать materialize inline

**Затронутые файлы:**
- `app/run_bot.py` — APScheduler setup
- `docker-compose.yml` — (опционально) отдельный сервис

### BUG-4: Превью расписания обрезано без кнопки «показать ещё» [MEDIUM]

**Корневая причина.**
`_render_preview_summary()` (`draft.py`) жёстко обрезает отображение до 8 записей: `preview.entries[:8]`. Оставшиеся показываются как текст `"... ещё N"` без интерактивных элементов.

**Импакт.**
Курсы бывают 42-60 дней с 20+ событиями. Пользователь видит только первые 8 записей и неинтерактивный текст «ещё 4». Невозможно просмотреть полное расписание.

**Решение.**
Реализовать пагинацию по неделям:
- Кнопка «📅 Полное расписание» открывает отдельную панель
- Панель показывает «окно» из 5-7 дней вокруг текущего дня
- Навигация: ◀️ Пред. неделя | ▶️ След. неделя
- Лимит Telegram — 4096 символов; для длинных курсов обязательна пагинация

**Затронутые файлы:**
- `app/bots/handlers/draft.py` — новая панель полного расписания с пагинацией

### BUG-5: Кнопка «Пересчитать preview» неуместна / вводит в заблуждение [MEDIUM]

**Корневая причина.**
`build_preview_actions()` (`draft.py`, строка 847) привязывает кнопку «Пересчитать preview» к `draft:calculate:run`, который **полностью пересчитывает** предпросмотр с нуля.

Проблема: та же клавиатура `build_preview_actions()` показывается и на панели «Course estimate» (после нажатия «Оценка курса»). На этой панели кнопка «Пересчитать preview» не имеет смысла — логичнее было бы «Показать предпросмотр» или «К расписанию».

**Решение.**
Сделать текст кнопки контекстно-зависимым:
- С панели предпросмотра: «🔄 Пересчитать»
- С панели оценки курса: «📋 К предпросмотру»
- С панели активного протокола: «📅 Показать расписание»

**Затронутые файлы:**
- `app/bots/handlers/draft.py` — `build_preview_actions`, `build_active_protocol_actions`

## 2. Отсутствующий функционал

### FEAT-1: Навигация «Назад» с панели оценки курса

**Текущее состояние.**
После нажатия «Course estimate» из предпросмотра или активного протокола, клавиатура содержит те же кнопки из `build_preview_actions()` или `build_active_protocol_actions()`. Кнопка «Назад» для возврата к предыдущему экрану отсутствует.

**Целевое состояние.**
Каждая панель должна заканчиваться единообразной строкой навигации:
```
[◀️ Назад] [🏠 Главная]
```
- «Назад» возвращает к предыдущему контексту (панель предпросмотра или панель протокола)
- «Главная» всегда ведёт на корневую домашнюю панель

Затронутые клавиатуры: `build_preview_actions`, `build_pre_start_actions`, `build_active_protocol_actions`, `build_draft_actions`, `build_settings_actions`, `build_wizard_navigation_actions`.

---

### FEAT-2: Полноценная панель просмотра расписания

**Текущее состояние.**
После расчёта пользователь видит компактный «Schedule preview» (первые 8 событий). После активации панель «Protocol activated» показывает только метрики — ни расписания, ни списка продуктов, ни таймлайна приёма.

**Целевое состояние.**
Отдельная панель **«План приёма»**, доступная из:
- Панели предпросмотра: кнопка «📅 Полное расписание»
- Панели активного протокола: кнопка «📅 График приёма»

Содержание панели:
- Текущий день цикла (например: «День 14 из 28»)
- «Окно» из 5-7 дней вокруг текущего (с пагинацией ◀️/▶️)
- Для каждого дня — все продукты с дозировкой (мг/мл)
- Для активных протоколов: ✅ для выполненных приёмов, ⬜ для ожидающих

---

### FEAT-3: Управление списком протоколов

**Текущее состояние.**
Пользователь может создать один протокол. Нет возможности:
- Просмотреть список всех протоколов (завершённых, активных, на паузе)
- Удалить или архивировать старые протоколы
- Реактивировать завершённый протокол

**Целевое состояние.**
Панель «Мой протокол» должна показывать:
- Активный протокол (если есть) с бейджем статуса
- Список предыдущих протоколов с датами и статусами
- Действия: Просмотр, Реактивация, Удаление

---

### FEAT-4: Завершение / пауза / возобновление протокола

**Текущее состояние.**
Метод `cancel_active_protocol()` существует в `draft_service`, но ни одна кнопка в UI его не вызывает. Нет действий «пауза» или «завершить».

**Целевое состояние:**
- «🏁 Завершить протокол» — когда курс пройден
- «⏸️ Пауза» — приостановка напоминаний с сохранением прогресса
- «▶️ Возобновить» — после паузы
- Автоматическая пауза / автозавершение при пропусках (см. FEAT-6)

### FEAT-5: Человекочитаемые уведомления-напоминания

**Текущее состояние.**
Текст напоминания генерируется в `_render_reminder_text()` (`app/application/reminders/service.py`):
```
Injection reminder: {event_key}
Product: {product_id}    ← UUID, не человекочитаемое имя
mg: N
ml: N
```

**Целевое состояние:**
```
⏰ Время приёма

💊 SP Sustanon FORTE 500
Доза: 250 мг (1.0 мл)
День 14 из 28

[✅ Принял]  [⏰ Позже]  [⏭️ Пропустить]
```

Для этого `_render_reminder_text()` должен получать display_name продукта (через join с таблицей `CompoundProduct`) вместо UUID.

### FEAT-6: Настройки автопаузы / автозавершения

**Целевое состояние.**
Добавить в панель Settings:
- «Автопауза после N пропусков» (по умолчанию: 3)

## 4. Карточка товара — медиа-презентация

**ID:** CARD-1
**Приоритет:** MEDIUM

### Текущее состояние

Файл: `app/bots/handlers/search.py`, функция `_render_product_card()` (строка 283).

Тогл «Show media» выводит в текст панели:
```
Media references
• ref #1
• ref #2
```
Это просто пронумерованные строки — ни изображений, ни видео, ни ссылок.

Тогл «Show sources» выводит текст «Ссылки доступны кнопками ниже» и формирует URL-кнопки из `card.media_refs[:3]` с подписями «Source 1», «Source 2», «Source 3». Смысл понятий "sources" и "media" перемешан.

Поле `media_refs` в `OpenCard` — это `list[str]`, содержащий:
- Внешние URL изображений (`https://...`)
- Telegram file_id в формате `tg-photo:XXXX`, `tg-video:XXXX`, `tg-animation:XXXX` (загруженные через админку)

Никакой реальной медиа-презентации нет: ни `answer_photo`, ни `InputMediaPhoto`, ни галереи.

### Целевое состояние (паттерн TradeFlow)

По аналогии с `shell_sku_media.py` из TradeFlow:

1. **Тогл «Медиа»** — при нажатии отправляется **отдельное сообщение** с фото/видео **над** текстовой панелью карточки
2. **Парсинг media_refs:**
   - `tg-photo:FILE_ID` → `bot.send_photo(photo=FILE_ID)`
   - `tg-video:FILE_ID` → `bot.send_video(video=FILE_ID)`
   - `tg-animation:FILE_ID` → `bot.send_animation(animation=FILE_ID)`
   - `https://...` → `bot.send_photo(photo=URL)` (Telegram сам загрузит)
3. **Навигация** — если медиа > 1: кнопки ◀️ Пред / ▶️ След
4. **Состояние** — в `CARD_STATE_KEY` добавить поля:
   - `media_preview_message_id` — ID отправленного медиа-сообщения
   - `media_index` — текущий индекс в массиве media_refs
5. **Очистка** — при закрытии карточки или переключении тогла на «скрыть» — удалить медиа-сообщение
6. **Источники** — отделить от медиа:
   - `official_url` → кнопка «🌐 Официальный сайт»
   - Все прочие media_refs из кнопок убрать (они теперь показываются как реальные медиа)

### Затронутые файлы

- `app/bots/handlers/search.py` — `_render_product_card`, `build_card_actions`, `on_toggle_section`, новые хендлеры media prev/next
- `app/bots/core/flow.py` — хелперы `send_media_preview()` / `delete_media_preview()`

### Технические детали

**Парсинг URL Google Sheets:**
```python
import re
match = re.search(r'/spreadsheets/d/([a-zA-Z0-9_-]+)', url)
sheet_id = match.group(1) if match else None
gid_match = re.search(r'[?#&]gid=(\d+)', url)
gid = gid_match.group(1) if gid_match else None
```

**Dry-run режим для `CatalogIngestService`:**
Добавить параметр `dry_run: bool = False` в метод `run()`. При `dry_run=True`:
- Выполнять все маппинги и валидации
- НЕ вызывать `upsert_product` и `record_source_row`
- Вернуть `IngestResult` с подсчётами

**Динамический `GoogleSheetsCatalogGateway`:**
Сейчас `sheet_id` и `tab_name` берутся из `GoogleSheetsConfig` при инициализации. Нужно поддержать передачу параметров при вызове `fetch_rows(sheet_id=..., tab_name=...)` для работы с динамической ссылкой.

## 8. Админ-панель — управление медиа (TradeFlow-паттерн)

**ID:** ADMIN-MEDIA
**Приоритет:** MEDIUM

### Текущее состояние

Файл: `app/bots/handlers/admin.py`, хендлер `on_media_input()` (строка 130).

Текущий флоу:
1. Админ нажимает «🖼️ Добавить медиа» на карточке продукта
2. Бот переходит в FSM-состояние `AdminMediaUploadState.waiting_input`
3. Админ отправляет фото/видео/URL
4. Создаётся `ProductMediaRef` с `ref_url` (например `tg-photo:FILE_ID`)
5. Показывается «✅ Медиа добавлено» + кнопка «Открыть карточку снова»

**Что отсутствует:**
- Нет Media hub — невозможно посмотреть, сколько медиа уже привязано
- Нет галереи — невозможно листать существующие медиа
- Нет «Установить основное» (primary media)
- Нет «Удалить» медиа
- Нет поддержки альбомов (множественных фото за раз)
- После загрузки приходится вручную возвращаться к карточке

### Целевое состояние (паттерн TradeFlow)

Эталонная реализация: `C:\Users\UraJura\TradeFlow\app\bot\shell_sku_media.py` (638 строк).

**Media Gallery панель:**

**Ключевые технические решения (из TradeFlow):**
- Per-chat lock для сериализации альбомных загрузок (`asyncio.Lock`)
- Debounce `0.6s` для обнаружения последнего фото в альбоме
- При переключении между медиа: удалить старый preview-message, отправить новый, затем переотправить панель НИЖЕ
- `auto-detect` режим: фото → photo, видео → video, URL → external link
